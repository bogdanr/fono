# Wave 3 Slice B1 Thread C — live Groq equivalence lane

**Status:** Draft, supersedes the cloud-mock approach in
`plans/2026-04-28-wave-3-slice-b1-v1.md` Tasks C1–C9 (now rejected;
see [Alternatives](#alternatives)).

## Objective

Close out Slice B1 by gating release tags on a real Groq round-trip
that exercises the cloud STT pipeline end-to-end, instead of mocking
it. The mock approach was attractive because it's offline and
deterministic, but it cannot detect the failure mode that matters
most: Groq silently changing their request/response shape, rate-limit
behaviour, or model defaults. A small, real, release-time check
catches that within minutes of tagging instead of minutes after a
user reports breakage.

## Why real beats mocked here

| Property | Mock | Real Groq |
|---|---|---|
| Offline / free | ✓ | ✗ (small cost — see budget) |
| Deterministic across CI re-runs | ✓ | mostly (Whisper output is stable on the same audio) |
| Catches our regressions | ✓ | ✓ |
| Catches Groq schema/behaviour changes | ✗ | ✓ |
| Maintenance cost (recording fixtures, refreshing them) | recurring | none |
| Risk if Groq is down at tag time | none | release blocked until Groq recovers (acceptable; we control retry) |

The maintenance cost of mock recordings was always going to bite —
every time Groq tweaks `verbose_json` or adds a field, the recordings
go stale and the gate gives a false signal. Real calls are
self-refreshing.

## Where the API key lives

GitHub Settings → Secrets and variables → Actions → New repository
secret:

- **Name:** `GROQ_API_KEY` (exact case)
- **Secret:** the full `gsk_...` token from
  https://console.groq.com/keys
- **Scope:** repository-level (Organization-level secrets are
  unnecessary for a single-repo project).

The key is referenced in the workflow as `${{ secrets.GROQ_API_KEY }}`.
GitHub does not expose repository secrets to workflow runs triggered
by PRs from forks, so this design is fork-safe by construction: the
cloud job sees an empty value on fork PRs and auto-skips.

## Budget shape

Groq's free tier for `whisper-large-v3-turbo` (verified as of late
2025; the plan re-checks at implementation time) is approximately:

- 7,200 audio-seconds per hour
- 28,800 audio-seconds per day
- 30 requests per minute

The plan's per-release budget is **30 audio-seconds × ~10 requests**
(5 fixtures × 6 s each, with one allowance for a rerun round-trip per
fixture). That is roughly 0.1 % of the daily audio cap. Even ten
releases per day stays under 2 %.

## Tasks

### C1. Add a real-Groq integration test feature flag

New Cargo feature `groq-live` on `crates/fono-stt`:

```toml
[features]
groq-live = []  # opt-in; enables tests gated on GROQ_API_KEY env var
```

New `crates/fono-stt/tests/groq_live.rs` test module, gated with
`#![cfg(feature = "groq-live")]`. Each test reads `GROQ_API_KEY` from
the env and **skips with a clear log line** (not fails) when absent
— so a developer running `cargo test --features groq-live` locally
without a key gets a tidy "skipping; set GROQ_API_KEY to enable"
message, not a confusing 401.

### C2. Commit ~5 short audio fixtures

Re-use the existing fixtures under `tests/fixtures/`. Pick 4–5 clips
that already have reference transcripts in the equivalence-harness
manifest, total ≤ 30 s. No new audio needs to be recorded.

The fixtures in scope are exactly the ones the existing
`fono-bench equivalence` runs against, so the new gate piggy-backs on
already-validated reference text and the `levenshtein_threshold`
already tuned per fixture in
`docs/bench/baseline-comfortable-tiny-en.json` (or a new sibling
baseline).

### C3. New baseline anchor for cloud rows

`docs/bench/baseline-cloud-groq.json` — sibling to the existing
`baseline-comfortable-tiny-en.json`. Same shape, with cloud-Groq
verdicts. Generated by running `fono-bench equivalence
--stt groq --baseline --no-legend` once locally, sanity-checked by
hand, and committed.

Verdicts must be **structural only** (skip / pass / fail), not exact
transcript matches — Whisper output occasionally differs by a token
even on identical audio. The Levenshtein gate already accommodates
this.

### C4. New `fono-bench equivalence --stt groq` mode

`crates/fono-bench/src/bin/fono-bench.rs:288-333` and `:659-684`:
add `groq` as a value alongside the existing `local` and
`cloud-mock`. Reuses the existing `Groq` client. No new wire format,
no new HTTP infrastructure.

Reads `GROQ_API_KEY` from env; if absent, exits cleanly with a
non-error message (so the workflow can decide whether absence is OK).

### C5. Rate-limit and 429 handling

The integration test code path:

- Sleeps 250 ms between fixtures (well under 30 req/min).
- Treats HTTP 429 as a **hard fail with an explanatory message** —
  not a retry — so a release that bumps into the cap gets a clear
  signal instead of looping. Recovery is "merge later or bump the
  cap"; not the workflow's problem to solve.
- Treats HTTP 5xx as transient: one retry after 2 s, then hard fail.
- Caps total wall time at 90 s.

### C6. Wire into `release.yml`, not `ci.yml`

A new `cloud-equivalence` job in `.github/workflows/release.yml`
runs **before** the `build` job's matrix expands:

```yaml
cloud-equivalence:
  name: Cloud STT equivalence (Groq)
  runs-on: ubuntu-latest
  if: ${{ secrets.GROQ_API_KEY != '' }}
  steps:
    - uses: actions/checkout@v5
    - uses: dtolnay/rust-toolchain@stable
    - uses: Swatinem/rust-cache@v2
    - name: Run live Groq equivalence
      env:
        GROQ_API_KEY: ${{ secrets.GROQ_API_KEY }}
      run: |
        cargo run --release -p fono-bench \
          --features equivalence,groq-live -- \
          equivalence \
            --stt groq \
            --output cloud-bench.json \
            --baseline \
            --no-legend
        python3 .github/scripts/diff-bench.py \
          cloud-bench.json \
          docs/bench/baseline-cloud-groq.json

build:
  needs: cloud-equivalence
  # ... existing matrix
```

The `if:` guard means a release tag pushed to a fork (or a contributor
without the secret configured) skips the cloud check rather than
failing it. Owner releases must pass it because the secret is set.

The `needs:` keeps the existing build/upload matrix from running if
the cloud check fails, so a broken release never gets artefacts.

### C7. Optional weekly canary (deferred — recommend, don't ship in C)

A second workflow `cloud-canary.yml` triggered on `schedule: cron`
once a week running the same gate. Catches "Groq changed something
between releases." Out of scope for closing Wave 3; flagged for a
follow-up task.

### C8. Manual override path

Operator escape hatch: pushing a tag with the suffix `-no-cloud-gate`
(e.g. `v0.3.0-rc1-no-cloud-gate`) skips the cloud-equivalence job.
Implemented as `if: ${{ ... && !endsWith(github.ref, '-no-cloud-gate') }}`.
Useful when Groq is down and a release truly cannot wait, or when
shipping a security fix that doesn't touch the STT path.

## Verification

1. With `GROQ_API_KEY` unset: workflow run on tag push skips
   `cloud-equivalence`, build job still runs, release artefacts still
   produced. (Reproduce by pushing a test tag from a clean fork.)
2. With `GROQ_API_KEY` set, all fixtures pass: workflow run completes
   green; release artefacts uploaded.
3. With `GROQ_API_KEY` set, one fixture intentionally broken (commit a
   bad reference transcript): cloud-equivalence fails, build job is
   skipped (no artefacts uploaded), tag remains in repo but no
   release object is created.
4. With `GROQ_API_KEY` set, Groq returns 429 (simulate by hammering
   the test locally): test exits with the explanatory message, not
   a retry loop.
5. Tag suffix `-no-cloud-gate`: cloud-equivalence job is skipped
   even with the secret set; build proceeds.

## Risks and mitigations

1. **Secret leak via misconfigured workflow.** The job uses `env:`
   to scope the secret to a single step. Risk: a malicious dependency
   in a build script could exfiltrate it. Mitigation: the cloud job
   runs *before* `build`, on a separate runner — and the `cargo
   build` it does is for `fono-bench` and `fono-stt` only, not the
   full workspace. Reduces the dependency surface materially.
   Stronger mitigation deferred: a dedicated minimal binary
   `groq-equivalence-runner` with its own pinned dep tree.
2. **Groq deprecates `whisper-large-v3-turbo`.** Detected by the
   gate failing; release blocked until the model name is updated in
   `crates/fono-stt/src/groq.rs`. Net positive — same failure would
   reach users otherwise.
3. **Free tier limits change.** Re-verify at implementation time;
   the budget headroom (≤ 0.2 % of daily cap) absorbs a 100×
   tightening before the gate becomes a problem.
4. **Test flakiness from Whisper output drift.** Mitigated by using
   the existing Levenshtein-distance gate, which already tolerates
   per-token jitter. If a specific fixture proves flaky after one
   month of releases, drop it and replace with a steadier clip.

## Alternatives

1. **Cloud-mock recordings (the rejected v1 approach).** Offline,
   deterministic, but cannot catch upstream changes. Maintenance
   recurring. See `plans/2026-04-28-wave-3-slice-b1-v1.md` Tasks
   C1–C9 for the original design.
2. **WireMock with hand-written JSON fixtures.** Same problem as 1,
   plus more lying — hand-written JSON drifts faster than recorded
   JSON.
3. **Real calls on every PR (not just releases).** Burns the free
   tier without proportionate benefit. Most PRs don't touch the
   cloud STT path. Release-time only is the right granularity.
4. **Run real calls *and* mock calls.** Doubles maintenance for
   marginal additional coverage. Defer; revisit only if real-call
   flakiness becomes a problem.

## Out of scope

- Cloud-mock infrastructure for OpenAI / other providers. Same
  argument applies — when those backends start gating release tags
  too, repeat this pattern with `OPENAI_API_KEY` etc.
- Streaming-mode equivalence. Slice B1 already shipped batch-vs-stream
  parity for the local backend; cloud streaming uses the
  pseudo-stream loop, which is exercised by the same fixtures the
  batch gate uses. No additional work.
- Per-PR cloud sniff test (deferred to weekly canary, C7).
