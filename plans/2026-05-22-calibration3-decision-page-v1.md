# calibration3 — reimagined model-decision visualisations

## Objective

Replace the current `docs/bench/calibration/summary/calibration.html`
(v1, generated by `scripts/bench-decision-page.py`) and
`docs/bench/2026-05-19-perf-pass/summary/calibration2.html`
(v2, generated by `scripts/bench-decision-page2.py`) with a single
new page `calibration3.html` that:

1. **Fixes every confirmed correctness bug** in v1 and v2 (enumerated
   below — currently the page can silently overwrite or invert data).
2. **Restructures the visual narrative** so each chart answers exactly
   one decision question, with consistent threshold semantics across
   every annotation.
3. **Preserves the data pipeline** — read from
   `docs/bench/calibration/summary/matrix.json` and the per-iteration
   accuracy JSONs in `docs/bench/calibration/runs/` (with optional
   merge from `docs/bench/2026-05-19-perf-pass/`). No bench reruns
   required; this is a presentation-layer rewrite.

The page is for the developer/operator audience (registry decisions,
key-findings document) AND for end-users picking a model on their own
hardware ("which whisper model can my laptop run live").

## Confirmed bugs to fix (catalogued from code + screenshot)

### v1 — `scripts/bench-decision-page.py`

- [ ] **B1. Family-key collapse conflates `.en` and multilingual cells.**
      `scripts/bench-decision-page.py:590-595` groups by
      `${host}/${build}/${model_family}` where `model_family` is
      `tiny|base|small|turbo` — so `base` and `base.en` share the
      same group key. Both write into `grouped[label][quant]`; the
      *last cell seen overwrites the first*. Visible in the
      screenshot at `i7-1255u/cpu/base` showing a single purple bar
      after a CPU quant sweep was committed — the q5_1/q8_0 cells
      exist in `matrix.json` but each `base/base.en` overwrites the
      other within the family bucket.
- [ ] **B2. Power-state collapse.** Same grouping ignores `power`,
      so `ac` and `battery` measurements overwrite each other (we
      have many `battery` runs in
      `docs/bench/calibration/runs/i7-1255u__battery__cpu__*`).
- [ ] **B3. Stale "key findings" text** at
      `scripts/bench-decision-page.py:275` asserts *"On CPU,
      quantization gives essentially zero speed benefit on every
      laptop CPU we measured."* This is now contradicted by the
      i7-1255u (Alder Lake, AVX-VNNI) CPU sweep — 3× to 9× speedups
      at q8_0/q5_1. Must be derived from data, not hardcoded.
- [ ] **B4. Empty quant slots render as visual gaps.** The bar
      chart leaves null slots for missing quants, indistinguishable
      from "measured but slow". Need an explicit
      "not-measured"/"n/a" visual or omit the slot entirely.
- [ ] **B5. Recommendation scorer at
      `scripts/bench-decision-page.py:518-527` ignores `power`** —
      battery cells can win the per-host best slot when they
      shouldn't.
- [ ] **B6. q5_0 legend slot is global** but only `large-v3-turbo`
      ships a q5_0 variant, so the swatch is misleading on every
      non-turbo group.

### v2 — `scripts/bench-decision-page2.py`

- [ ] **B7. Inverted RTF threshold annotations.** Frontier scatter
      annotations at `scripts/bench-decision-page2.py:626-631`
      label `RTF=2` as *"2× (borderline)"* and `RTF=3` as
      *"3× (unsuitable)"*. The verdict definition in
      `scripts/bench-summarise.py:194-198` is `batch ≥ 2.0 AND
      stream ≥ 1.5 → comfortable`; lower RTF is *worse*, not the
      reverse. The page labels imply the opposite — RTF=3 is in
      fact great.
- [ ] **B8. Inverted capability-ladder annotations.**
      `scripts/bench-decision-page2.py:697-700` labels
      *"comfortable (<2×)"* and *"borderline (<3×)"*. Stream RTF
      ≥ 1.5 is the comfortable floor; *< 2×* is the borderline
      region.
- [ ] **B9. Inverted Vulkan-chart Y-axis label.**
      `scripts/bench-decision-page2.py:947` —
      *"Batch RTF (lower = faster)"*. Higher RTF is faster.
- [ ] **B10. Inverted Vulkan speedup tooltip.**
      `scripts/bench-decision-page2.py:939` computes
      `cpu/vk` and labels it *"Speedup"*. If Vulkan is faster
      (vk > cpu) this ratio is < 1, which is the inverse of
      "speedup". Correct ratio for "Vulkan speedup vs CPU" is
      `vk/cpu`.
- [ ] **B11. Subtitle threshold text** at
      `scripts/bench-decision-page2.py:361` says
      *"<2× is comfortable"*. Comfortable floor is stream ≥ 1.5,
      not stream < 2. Wording is ambiguous at best and inverted
      under the most natural reading.
- [ ] **B12. `model_family` `find()` order dependence.**
      Capability ladder
      (`scripts/bench-decision-page2.py:668-670`), TTFF scatter
      (`scripts/bench-decision-page2.py:723`), Vulkan chart
      (`scripts/bench-decision-page2.py:909-915`) all use
      `RAW_CELLS.find(...)` keyed on `model_family` which
      collapses `base` ↔ `base.en` and `ac` ↔ `battery`. Whichever
      cell happens to be first in serialization order wins.
- [ ] **B13. Vulkan-chart host bug.** The host lookup at
      `scripts/bench-decision-page2.py:909-915` uses
      `showHosts.find(h => RAW_CELLS.some(c => c.host===h && c.build==='cpu' && c.model===m))`
      to pick a host for each model — when multiple hosts qualify,
      the *first* host wins, even though the chart legend implies
      a single fixed host filter.
- [ ] **B14. "Best cell" verdict heatmap can prefer quant over
      fp16 baseline** because the scorer ties on verdict and then
      picks lowest RSS. This hides the fp16 baseline performance
      a registry default needs to express.
- [ ] **B15. Disk-vs-accuracy averages accuracy across hosts**
      without surfacing host variance. Acceptable, but currently
      undocumented in the legend, and same family color makes
      adjacent quant points indistinguishable.

### Generator hygiene

- [ ] **B16. `scripts/bench-summarise.py` writes one cell per
      `(host, power, build, model)`** (good), but the v1/v2 page
      generators discard `power` before grouping. The fix is in
      the pages, not the summariser.
- [ ] **B17. matrix.json has no schema validation step.** A
      smoke-test that asserts uniqueness of
      `(host, power, build, model)` per cell, and presence of
      `iterations_kept ≥ 1`, would catch malformed merges
      (perf-pass ∪ calibration) early.

### 3. Visualisation grammar

The page presents eight charts plus a header banner and a full
filterable table. Every chart obeys these invariants:

- Higher RTF = faster (consistent across all axes & annotations).
- Thresholds drawn at `batch_rtf=2.0` and `stream_rtf=1.5` are
  labelled with **direction** ("≥ 2× = comfortable",
  "< 1× = unsuitable").
- A cell with `iterations_kept < 2` is rendered with a striped
  fill and excluded from "best cell" computations.
- Tooltips always include: host, power, build, model, lang,
  quant, batch RTF, stream RTF, TTFF, RSS MiB, Lev, iterations.

- [ ] **Task 3.1. Chart A — Decision Heatmap.**
      Rows = hosts (sorted by `host_year` ascending). Columns =
      `(family × language)` cross-product, in order
      tiny / tiny.en / base / base.en / small / small.en / turbo.
      Each cell is the *recommended* quant for that
      (host, family, language) at AC power, CPU build, sourced
      by a scoring function that prefers `comfortable` verdict,
      smallest RSS within a verdict band, and `accuracy_pass`
      true. The cell shows the quant name, stream RTF, RSS MiB,
      and a ⚡ badge if Vulkan improves the verdict for that
      same (host, family, language). Fixes B1, B2, B5, B14.
- [ ] **Task 3.2. Chart B — Speed Sweep (faceted small
      multiples).** One panel per host. X = model variants
      grouped by family then language. Within each variant, three
      bars at most: fp16 / q8_0 / q5_1 (q5_0 only appears in the
      `turbo` cluster). Y = batch RTF log-scale. Threshold band
      shaded at y=1.0 and y=2.0. Backend (cpu / vulkan) is a tab
      or radio above the panels — never mixed in the same panel.
      Fixes B1, B2, B4 (explicit empty slots labelled "n/a"),
      B6 (q5_0 only on turbo).
- [ ] **Task 3.3. Chart C — Quant Speedup vs fp16.**
      Bar chart. X = (host, family, language). Y = ratio of
      quant batch RTF / fp16 batch RTF, log-scale. Two bars per
      x-tick (q8_0, q5_1) plus a horizontal line at 1.0 (no
      speedup) and 2.0 (textbook VNNI ceiling). Hosts grouped
      visually by `host_quant_kernel`: VNNI bloc on left,
      avx2-fallback bloc on right, separator. Surfaces the
      VNNI-vs-pre-VNNI story directly. Fixes B3 (the "no
      speedup" claim was correct for pre-VNNI hosts, wrong for
      VNNI hosts; this chart shows the dichotomy).
- [ ] **Task 3.4. Chart D — Accuracy vs Speed Frontier.**
      Scatter. X = batch RTF (log). Y = mean EN Levenshtein
      (lower better, axis *inverted* visually so up-right is
      strictly better). Point shape encodes quantization
      (circle=fp16, square=q8_0, triangle=q5_1, diamond=q5_0).
      Point colour encodes host. Pareto frontier line drawn.
      Threshold annotations at x=2.0 (comfortable floor),
      y=0.05 (per-fixture acc ceiling per
      `scripts/bench-summarise.py:194-198` discussion).
- [ ] **Task 3.5. Chart E — CPU vs Vulkan paired bars.**
      For each (host, family, language, quant) present in BOTH
      cpu and vulkan, draw a paired bar: cpu_rtf, vulkan_rtf.
      Speedup label uses `vk / cpu` (correct direction). Y log.
      Fixes B9, B10.
- [ ] **Task 3.6. Chart F — TTFF vs Stream RTF.**
      Scatter. X = TTFF seconds (lower better). Y = stream RTF
      (higher better). Annotations: x=0.5 (good live latency),
      y=1.5 (comfortable stream floor). Encoding: host=colour,
      family=shape, quant=opacity. Helps the "live dictation
      feel" decision that the verdict alone hides.
- [ ] **Task 3.7. Chart G — Disk vs Accuracy.**
      Bar chart per model (X), two stacked Y axes: bar height =
      mean Lev, secondary axis = approx download MiB. Coloured
      by family, hatched by quant. Replaces v2's scatter, which
      had indistinguishable points.
- [ ] **Task 3.8. Chart H — Coverage Matrix (diagnostic).**
      Grid. Rows = hosts. Columns = (power × build × language ×
      quant). Cells colour-coded: green = ≥2 iterations, yellow
      = 1 iteration, grey = not measured, red = errored. Lets
      the user see "what's missing" at a glance (i7-7500u CPU
      quant, ultra7-258v CPU quant, all CPU turbo quant, etc.).

## Potential Risks and Mitigations

1. **Chart.js plugin annotation drift.** v2 already depends on
   `chartjs-plugin-annotation@3.1.0` from CDN; v3 should pin
   the same major version and degrade gracefully if the plugin
   fails to load (annotations are nice-to-have; chart core is
   essential).
   *Mitigation:* feature-detect `Chart.registry.plugins.get('annotation')`
   before adding annotations; skip silently if absent.
2. **Filter state explosion.** With ~7 filter dimensions, naïve
   re-rendering on every change can stall on 200+ cells.
   *Mitigation:* compute filtered cells once per change and
   pass to all chart builders; charts re-use Chart.js update
   APIs rather than `destroy() + new Chart()`.
3. **Pre-VNNI hosts produce noisy quant ratios.** When fp16 and
   q8_0 take the same wall time within noise, the ratio chart
   can show ratios just above or below 1.0 that mislead.
   *Mitigation:* annotate any bar whose 95 % CI crosses 1.0 as
   "no significant speedup" (compute CI from
   `batch_rtf_stddev_pct` already in matrix.json).
4. **Generated HTML grows large.** v2 is already 200+ KB
   embedded. Adding coverage matrix + small multiples pushes
   to 400+ KB.
   *Mitigation:* trim to only `js_fields` the JS actually
   reads (v2 already does this at
   `scripts/bench-decision-page2.py:1095-1103`); keep the
   same discipline.
5. **Inventory schema drift.** Some hosts may lack
   `has_avx_vnni`, `released`, or `cpu_tier`.
   *Mitigation:* `validate()` warns (does not fail) on
   missing inventory fields; charts fall back to "?" labels.
6. **User confuses calibration1/2/3.** Keeping all three
   pages alive risks linking the wrong one.
   *Mitigation:* Task 8.1 banner + Task 8.3 link audit.
7. **Battery cells already merged silently into `comfortable`
   verdict statistics.** The verdict in `bench-summarise.py`
   doesn't distinguish power state, so cross-power comparisons
   may be off.
   *Mitigation:* filter-default to `power=ac`; document the
   caveat in the page's "About this data" footer.

## Alternative Approaches

1. **Patch v1 / v2 in place instead of writing v3.**
   *Trade-off:* faster; preserves URLs; but the bugs are
   structural (family-collapse, axis inversions) and
   substantive patches touch most of the JS — effectively a
   rewrite, with extra cost of preserving backwards
   compatibility. Rejected on net.
2. **Switch from Chart.js to D3/Observable Plot.**
   *Trade-off:* better small-multiples and faceting; much
   richer scatter customisation. But adds a build-step
   dependency (Plot is ESM-only on CDN, more fragile than
   Chart.js UMD) and is a larger lift. Rejected for v3;
   reconsider for v4 if Chart.js bar facets prove too
   limiting.
3. **Move presentation server-side (e.g. a Rust binary that
   serves an interactive page).**
   *Trade-off:* could share types and constants with
   `crates/fono-stt/src/registry.rs` directly. But the
   benchmark page is meant to be a single offline HTML file
   that ships in `docs/`. Server-side is out of scope.
4. **Render multiple smaller HTML pages (one per host).**
   *Trade-off:* simpler per-page logic; no global filter
   bar. Loses the cross-host narrative that's the entire
   point of the calibration matrix. Rejected.
5. **Generate static SVG charts instead of Chart.js.**
   *Trade-off:* no JS dependency, prints cleanly. But loses
   filter interactivity, which is the most-used feature
   on v1 right now (per the screenshots in this session).
   Rejected.
