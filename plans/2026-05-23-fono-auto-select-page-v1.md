# Fono auto-select page — feature-driven exploration tool + policy export

## Objective

Build a new HTML page,
`docs/bench/calibration/summary/auto-select.html`, generated by a new
`scripts/bench-auto-select-page.py`, that:

1. Exposes the calibration matrix as an **interactive data
   exploration tool** keyed on host features Fono can reliably detect
   at startup (CPU flags, released year, core count, RAM, GPU
   presence, binary variant, architecture). Crucially **not** keyed
   on coarse host classes ("laptop_modern", "desktop") — those
   abstract away the variables we actually care to learn about.
2. Computes a per-host recommendation walk under live,
   user-adjustable thresholds (no safety multipliers; thresholds are
   already arbitrary and surfaced as sliders) and absolute accuracy
   gates (not delta-vs-fp16).
3. Emits a `auto-select-policy.json` blob captured at the current
   slider state, which a future `crates/fono-stt/src/auto_select.rs`
   (replacing the `LocalTier::default_whisper_model()` path in
   `crates/fono-core/src/hwcheck.rs:77-83`) consumes at compile time
   to drive runtime model selection.

The page is **decision support**, not a frozen policy view. We expect
several rounds of "tweak sliders → see which models qualify → bake
into Rust" before the policy stabilises. The page outlives that
stabilisation as the regression-tracking surface when new hosts or
new models are added to the matrix.

## Why this shape and not a host-class table

User direction (paraphrased): host-class buckets ("laptop_modern_vnni"
etc.) discard the very features we want to learn about. The right
question is *which* host attributes actually predict per-model
viability — and that becomes a property of the cells, not a property
of the row aggregation. The page should let the human notice patterns
the data hides (e.g. "quantization gives zero uplift on Vulkan, on
every host we measured") rather than force the human to confirm a
pre-baked taxonomy.

The current `hwcheck.rs::tier()`
(`crates/fono-core/src/hwcheck.rs:175-198`) is wrong in exactly that
way: it dispatches on `cores + ram + disk` alone, ignores AVX-VNNI,
ignores quant choice, treats "accelerated" as a single boolean with a
flat 4× multiplier (`crates/fono-core/src/hwcheck.rs:251-256`), and
the per-model `realtime_factor_cpu_avx2` baked at
`crates/fono-stt/src/registry.rs:181-183` is a single number from one
host on one date. Closing that gap is the long-term payoff of this
work.

## Architecture handling (x86_64 today, aarch64 + Apple Silicon next)

All 5 hosts in `docs/bench/calibration/summary/matrix.json` are
x86_64. We will not have ARM or Apple Silicon measurements before
the first cut of this page. Design the page and the policy JSON
schema such that:

- Every cell carries an explicit `arch ∈ {x86_64, aarch64,
  apple_silicon}` field, populated from the inventory.
- CPU-flag fields are arch-aware: on x86_64 the page reads
  `avx2`, `avx_vnni`, `avx512`, `avx512_vnni`; on aarch64,
  `neon`, `sve`, `sve2`; on Apple Silicon, flags collapse to a
  single `apple_amx` boolean for now (Metal/CoreML acceleration
  is via `gpu_present` rather than a CPU flag).
- The exploration page's controls include an `arch` filter
  pinned to `x86_64` by default (since that is the only arch
  with data). The policy JSON the page emits **must** include
  explicit fallback rows for `aarch64` and `apple_silicon`
  saying "no calibration data yet — use conservative defaults"
  so the runtime classifier never panics on ARM.
- Binary variant becomes the cross product of arch × backend
  capability: `x86_64-cpu`, `x86_64-vulkan`,
  `aarch64-cpu`, `aarch64-vulkan`, `apple_silicon-metal`. The
  page's "Binary variant" toggle is gated to combinations that
  exist for the active arch filter.

Doing arch up front is cheaper than retrofitting; the schema cost
is a couple of additional fields per cell and a hard-coded
fallback table in the policy JSON.

## Implementation Plan

- [ ] Task 1. Add a `derive_host_features(host_id, inventory)`
      helper in `scripts/bench-auto-select-page.py` that emits, per
      host: `arch`, `released_year`, `physical_cores`, `ram_gb`,
      `cpu_flags: {avx2, avx_vnni, avx512, avx512_vnni}` for x86_64,
      `cpu_model_str` (for the human reading the page only, never
      consumed by the policy), `gpu_present: bool`,
      `gpu_class ∈ {none, integrated, discrete, apple_metal}`,
      `gpu_model_str`. Reads from
      `docs/bench/calibration/inventory/*.json`. Rationale: the
      features become first-class columns on every cell so the
      page's scatter views, filters, and policy walk all read the
      same surface. Mirror the helper as a Rust function in a
      follow-up task so the runtime classifier sees the same field
      names and types as the page; out of scope for this plan but
      earmarked.

- [ ] Task 2. Extend the cell-enrichment step (modeled on
      `scripts/bench-decision-page3.py:160-190`) to attach `arch`,
      `cpu_flags`, `gpu_class`, `released_year`, `cpu_model_str` to
      every cell. Drop the `host_quant_kernel` derived label from
      this output — the page will compute its own groupings on the
      fly from `cpu_flags.avx_vnni`. Surface the absolute accuracy
      `accuracy_en_mean` and `accuracy_all_mean` from the existing
      accuracy loader without computing or attaching a `delta_*`
      companion (per user direction: absolute accuracy is primary).

- [ ] Task 3. Build the page skeleton in
      `scripts/bench-auto-select-page.py` mirroring the dark theme
      and embedding model of `bench-decision-page3.py`
      (`scripts/bench-decision-page3.py:280-490`). Single page,
      sticky filter bar at top, sections beneath. Embed the
      enriched cell array, the inventory, and the threshold
      defaults as `RAW_CELLS`, `HOST_FEATURES`, `THRESH_DEFAULTS`
      JS literals. Pin Chart.js 4.4.3 + `chartjs-plugin-annotation`
      from jsdelivr (same CDN sources used today in
      `scripts/bench-decision-page3.py:298-299`).

- [ ] Task 4. Implement the control bar with eight live controls.
      URL-hash backing identical in style to
      `scripts/bench-decision-page3.py:534-557`. Controls and
      defaults:
      - **Batch RTF threshold** — number input, default 2.0,
        range 1.0–4.0 step 0.1.
      - **Absolute accuracy ceiling** — number input, default 0.10,
        range 0.00–0.30 step 0.01, label "Levenshtein CER
        (lower = better)".
      - **Stream RTF soft floor** — number input, default 1.5,
        range 0.8–3.0 step 0.1. Secondary; only used to add a
        warning chip, never to disqualify.
      - **Memory budget MiB** — number input, default 1024,
        range 256–4096 step 128. Hard gate on
        `peak_rss_mib_median`.
      - **Binary variant** — toggle: `cpu_only` / `gpu_capable`.
        Default `gpu_capable`. Filters cells by `build`:
        `cpu_only` drops `build='vulkan'` rows, `gpu_capable`
        keeps both and lets the policy prefer one.
      - **Language requirement** — toggle: `english_ok` /
        `multilingual_required`. Default `english_ok` (wider, per
        user direction). When `multilingual_required`, `.en` model
        variants are removed from the candidate set.
      - **Quant preference** — select: `auto`, `force_fp16`,
        `force_q8`, `force_q5`. Default `auto`.
      - **Power** — select: `ac`, `battery`, `both`. Default `ac`.
      - **Arch** — select pinned to `x86_64` for v1 with a
        tooltip explaining "aarch64 / Apple Silicon: no
        calibration data yet — fallback rules used in policy
        JSON output".

- [ ] Task 5. Implement the per-host **recommendation walk** as
      the headline section. Algorithm:
      ```text
      candidates = cells matching all hard gates:
        host == this_host
        power matches the power control
        build is in the set allowed by the binary-variant toggle
        language is in the set allowed by the language-requirement toggle
        quant is in the set allowed by the quant-preference toggle
        batch_rtf_median >= batch_rtf_threshold
        accuracy_en_mean <= accuracy_ceiling
        peak_rss_mib_median <= memory_budget_mib
      pick = the candidate maximising preference order:
        1. largest family    (turbo > small > base > tiny)
        2. fp16 > q8 > q5    (more accurate within a family)
        3. backend tiebreak: prefer vulkan iff its batch RTF >=
           1.2 * the same model's cpu batch RTF on this host
        4. iterations_kept descending (statistical confidence)
      ```
      Per host the page renders:
      - chosen tuple as a chip: `family / language / quant /
        backend`;
      - one-line trace ("trace" because the user must be able to
        audit *why*): the gate values the pick passed plus the
        gate values the next-up alternative would have failed;
      - a stream RTF warning chip if `stream_rtf_median <
        stream_rtf_soft_floor`;
      - a coverage chip showing `iterations_kept/total` and a
        red/amber dot when `iterations_kept < 2` or `errors[]`
        non-empty.
      Rationale: this is the headline output. Everything below
      exists to justify it.

- [ ] Task 6. Implement the **feature-vs-outcome scatter grid**:
      a 2×3 lattice (cores / ram_gb rows × batch_rtf / accuracy /
      peak_rss columns). Each dot = one cell. Encoding:
      - **Color** by `cpu_flags.avx_vnni` (binary: VNNI / no-VNNI).
      - **Shape** by quant (○ fp16 / ■ q8 / ▲ q5).
      - **Size** by family (tiny→small dot, turbo→large dot).
      - **Tooltip** shows the full cell descriptor.
      Use Chart.js scatter type. Add a horizontal threshold line
      on the `batch_rtf` column (driven by the slider) so the user
      can see at a glance which dots qualify. Rationale: this is
      where "data hides things we don't expect" becomes visible.

- [ ] Task 7. Implement the **quant uplift per host** view
      (reframe of the current calibration3 chart 3 but with a
      different question). One chart, x-axis = host. For each host
      compute `median(quant_rtf / fp16_rtf)` across all measured
      models, separately for `build = cpu` and `build = vulkan`.
      Bar height = the median. Label each bar with a categorical
      tag derived from the value: `large (>=1.5×)` / `moderate
      (1.1–1.5×)` / `none (0.9–1.1×)` / `regression (<0.9×)`.
      Caption documents the rule of thumb that falls out: "on
      Vulkan, every host we measured lands in `none` or
      `regression` — Vulkan-capable binaries should force fp16 at
      auto-select time". Rationale: the chart now has a single
      thesis it tests rather than a 70-bar wall.

- [ ] Task 8. Implement the **Pareto frontier per host** view.
      One scatter per host, x = `accuracy_en_mean` (lower better,
      reverse axis), y = `batch_rtf_median` (higher better). Each
      dot = one candidate `(family, lang, quant, backend)`.
      Compute and highlight the Pareto frontier (no other dot is
      both more accurate AND faster). Mark the current
      recommendation with a target reticle. Mark the user's
      accuracy ceiling and batch threshold as dashed rule lines
      so the qualifying quadrant is visually obvious. Rationale:
      makes the trade-offs the recommendation is implicitly
      choosing visible.

- [ ] Task 9. Implement the **policy JSON export** section. A
      button "Copy policy as JSON" emits a blob like
      ```json
      {
        "schema_version": 1,
        "generated_at": "2026-05-23T15:30Z",
        "thresholds": { "batch_rtf": 2.0, "accuracy_ceiling": 0.10, ... },
        "constraints": { "binary_variant": "gpu_capable",
                         "language_requirement": "english_ok", ... },
        "rules": [
          { "arch": "x86_64", "cpu_flags.avx_vnni": true,
            "cores_min": 4, "ram_gb_min": 4,
            "binary_variant": "gpu_capable",
            "recommend": { "family": "turbo", "language": "multi",
                           "quant": "fp16", "backend": "vulkan" },
            "evidence_hosts": ["i7-1255u","ultra7-258v"],
            "evidence_iterations_min": 2 },
          { "arch": "x86_64", "cpu_flags.avx_vnni": false, ... },

            "recommend": { "family": "base", "language": "multi",
      ```

      - **Missing measurements**: enumerate the candidate set

      - **Single-iteration cells**: any cell with

        recommendations resting on n=1 — flag in amber.
      - **Missing accuracy**: any cell selected by the walk where
        `accuracy_en_mean` is null. The policy cannot honour the

## Verification Criteria

- `scripts/bench-auto-select-page.py` exits 0 on the current
  matrix, writes `docs/bench/calibration/summary/auto-select.html`
  with all measured cells embedded.
- The page renders without console errors in a modern Chromium.
- Each of the eight controls updates every section live;
  URL-hash round-trips on reload.
- The recommendation walk shows a chosen tuple for every measured
  host under default settings; for hosts where no candidate
  satisfies the gates, the walk explicitly says "no candidate"
  and cites which gate killed which next-up candidate.
- The scatter grid colors are obviously bimodal on
  `cpu_flags.avx_vnni` in the batch-RTF column for CPU build
  cells, and obviously flat (no separation) for Vulkan-build
  cells. (Verifies the visualization actually shows the kernel
  effect.)
- The quant uplift chart labels at least one Vulkan host as
  `none` or `regression`. (Verifies the chart surfaces the
  "Vulkan + quant = no gain" finding the data already supports.)
- The Pareto chart for at least one host shows ≥3 candidates on
  the frontier, and the recommendation reticle is on a frontier
  point.
- The **Data Gaps** section renders at the bottom of the page,
  collapsed by default; opening it lists at least one entry per
  arch-fallback (`aarch64`, `apple_silicon`) and at least one
  copy-to-clipboard re-bench command for any
  `iterations_kept < 2` cell currently being picked. The footer
  one-liner shows `gaps under current policy: G` with the count
  updating live as filters change.
- The policy JSON button copies a syntactically valid JSON blob
  with `schema_version: 1` and rules for every distinct
  `(arch, cpu_flags.avx_vnni)` combination present in the matrix,
  plus the two hard-coded `_fallback: true` entries for
  `aarch64` and `apple_silicon`.

## Potential Risks and Mitigations

1. **The page reveals the matrix is too thin to derive a robust
   policy.** With only 5 hosts (2 VNNI laptops + 2 non-VNNI
   laptops + 1 desktop) the rule-merging step in Task 9 will
   collapse to essentially the same two policies the host-class
   approach would have produced. Mitigation: that's a finding,
   not a defect. The page should banner "n=N hosts measured per
   rule" prominently and the policy JSON should carry
   `evidence_hosts` so the human can see when a rule rests on one
   machine. Adding more hosts to the matrix (Zen4 desktop,
   AVX-512 server, an aarch64 SBC, an Apple Silicon Mac) is the
   obvious follow-up but is outside the scope of this page.

2. **The exploration UX outgrows Chart.js.** Eight live controls
   driving five chart views with cross-filtering is on the edge
   of what plain Chart.js + per-call rebuild can do smoothly with
   237 cells. Mitigation: keep each chart self-contained (each
   one calls `filteredCells()` and rebuilds independently, same
   pattern as `scripts/bench-decision-page3.py:980-987`), and
   accept ~50 ms re-render per slider tick on the current matrix.
   If it lags at 500+ cells we can incrementally lift only the
   affected charts.

3. **The policy JSON schema gets locked in too early.** Adding
   v1 to `schema_version` is a commitment. Mitigation: keep the
   schema minimal in this draft (arch, cpu-flag predicates,
   cores floor, ram floor, recommended tuple, evidence) and
   explicitly document that "rule merging" between page output
   and Rust consumption is manual in v1; future versions can
   introduce compound predicates without changing schema version
   if we stick to additive fields.

4. **Arch is hard to detect on first-run on Linux.** Apple
   Silicon reports `arm64` from `uname -m` but `apple_silicon`
   is the right tag because the Metal/CoreML backend is
   fundamentally different. Mitigation: the runtime classifier
   will use a combined `(os, arch)` discriminator —
   `(macos, arm64) → apple_silicon` — but this page only deals
   with one arch in v1 and can punt the multi-axis discriminator
   to the follow-up Rust task.

5. **Accuracy is sample-set-specific.** `accuracy_en_mean` is
   the Levenshtein character error rate on whatever English
   fixtures we benched. Real-world user dictation may have
   higher or lower WER depending on background noise, accent,
   vocabulary. Mitigation: surface this caveat in the page's
   caption text and include the per-cell `iterations_kept` so
   the user can judge confidence; do not present the absolute
   accuracy number as if it is the user's expected WER.

## Alternative Approaches

1. **Skip the new page entirely and bake a hand-authored policy
   in `hwcheck.rs` directly.** Trade-off: fastest path to a
   fixed `recommend_local_model()`, but loses the iterative
   tuning loop that the slider-driven page enables and the
   regression-tracking surface for new hosts/models. Rejected —
   without the page we are guessing what to put in Rust.

2. **Auto-merge per-host rules into a decision tree
   (sklearn-style induction) at policy-export time.** Trade-off:
   removes the manual rule-merge step and emits cleaner policy
   JSON. Worth doing once we have ≥20 measured hosts; with 5
   hosts the induced tree would just memorise the matrix.
   Deferred to a future iteration; v1 emits one rule per host
   plus an explicit `_evidence_hosts` field so the human knows
   what to merge.

3. **Reuse `bench-decision-page3.py` and add the new sections
   as extra tabs.** Trade-off: less code to maintain but
   conflates the diagnostic page (calibration3) with the policy
   page (auto-select). The user explicitly asked for a separate
   file; keeping the two scripts and the two HTML files lets
   each evolve without breaking the other.

4. **Use a Python notebook (Jupyter) for the exploration step
   and only ship a frozen policy JSON.** Trade-off: notebooks
   are ergonomic for slicing and dicing but cannot be shipped as
   a browsable artefact in the repo, do not run in the user's
   browser, and bias the team toward one Python-fluent person
   maintaining the analysis. Rejected — the page belongs on
   `docs/bench/calibration/summary/` next to its siblings so
   anyone reviewing a calibration session can audit it.
