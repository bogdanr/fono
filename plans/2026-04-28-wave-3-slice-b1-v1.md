# Wave 3 — Slice B1: realtime cpal-callback push + Groq streaming + equivalence cloud rows

## Objective

Land the three pieces of Slice B1 from the strategic plan, mapping
to v1/v2/v3-vN of the interactive plan as follows:

1. **Realtime cpal-callback push** (Thread A) — replace the
   poll-a-buffer-every-30ms drain task at
   `crates/fono/src/session.rs:840-883` with a true cpal-callback push
   path. cpal's audio thread forwards PCM directly into the live
   `Pump` via a lock-free channel; the existing `RecordingBuffer`
   mutex hot-path is removed from the live trail entirely. Targets
   `live.first_partial` ≤ 400 ms (R10.1 telemetry; no telemetry work
   in this slice — measured manually).
2. **Groq streaming STT backend** (Thread B) — R4.2 from
   `plans/2026-04-27-fono-interactive-v1.md:75-77`. Implements
   `StreamingStt` via a "pseudo-stream" loop that re-POSTs the
   trailing N seconds of buffered audio to Groq's existing batch
   endpoint every ~700 ms, pipes results through the existing
   `LocalAgreement` helper, emits `Preview` updates, and emits
   `Finalize` on `SegmentBoundary` / `Eof`. In-flight request
   cap = 1 (drop overlap). Selectable via `fono use stt groq` with
   `[interactive].enabled = true` and `[stt.cloud].streaming = true`.
3. **Equivalence harness cloud rows** (Thread C) — R18.12 from
   `plans/2026-04-27-fono-interactive-v6.md:129-134`. New
   `--stt cloud-mock` mode in `fono-bench equivalence` that swaps
   the real Groq client for a recorded-HTTP mock so CI runs offline.
   Drops 1–2 cloud rows into the baseline JSON anchor at
   `docs/bench/baseline-comfortable-tiny-en.json` (or sibling).

This plan **modifies Rust source code**. Three DCO-signed commits
expected, one per thread. Threads are independent except C depends
on B (the mock targets the real Groq streaming backend).

This plan unlocks the v0.3.0 release train. Wave 4 (auto-translation)
ships next; Slice B2 (overlay subprocess) and Slice B3 (R18 Tier-2 +
fixture completion) are deferred to a later wave.

## Background — what is already in place

### Thread A — live capture path

Already shipped (`crates/fono/src/session.rs:723-901`,
`crates/fono/src/live.rs:420-460`):

- `Pump` owns an `AudioFrameStream` + `WebRtcVadStub`; `push(&[f32])`
  feeds chunks into the VAD-gated broadcast (`live.rs:438-440`).
- `LiveSession::run` consumes `FrameEvent`s from the broadcast and
  produces `LiveTranscript` (`live.rs:137`).
- Session orchestration spawns three tasks per live session
  (`session.rs:780-883`):
  1. **cpal capture thread** — calls `AudioCapture::start()` which
     internally writes raw PCM into a shared `RecordingBuffer`
     (`crates/fono-audio/src/capture.rs`). Capture thread blocks on
     `capture_stop_rx.recv()` until the session ends.
  2. **drain task** — polls `buffer.lock()` every 30 ms, slices off
     the new tail, calls `pump.push(tail)` (`session.rs:846-882`).
  3. **run task** — `LiveSession::run` consuming the broadcast.

Still missing (this thread's work):

- The 30 ms polling tick is the floor on every PCM sample's path
  through to `Pump`. It also forces a `Mutex` round-trip per tick.
  The cpal callback already produces frames at hardware cadence
  (~10 ms at 16 kHz / 160 sample buffers); the right shape is a
  lock-free SPSC channel from cpal callback → tokio task →
  `Pump::push`. No 30 ms tick, no `RecordingBuffer` middleman for
  live sessions.

### Thread A — Realtime cpal-callback push

#### Phase A1 — Capture-side API

- [x] Task A1. Add a sibling constructor / start method to
  `crates/fono-audio/src/capture.rs` along the lines of:
  ```rust
  /// Start capture wired to a real-time PCM forwarder. Each cpal
  /// data-callback invocation pushes its `&[f32]` slice through
  /// `forward(...)` directly. Differs from [`Self::start`] which
  /// accumulates into a [`RecordingBuffer`] for batch use.
  pub fn start_with_forwarder<F>(&self, mut forward: F) -> Result<CaptureStreamHandle>
  where
      F: FnMut(&[f32]) + Send + 'static,
  ```
  Returns a `CaptureStreamHandle` that owns the cpal `Stream` and
  drops it on `drop` (matches the existing `CaptureHandle` shape
  but without the `RecordingBuffer`). Sample-rate conversion (the
  resampler at `crates/fono-audio/src/resample.rs`) MUST run in the
  callback path so `forward(&[f32])` only ever sees mono f32 at
  `target_sample_rate`. If that's not feasible inside the
  realtime callback (pre-existing resampler is allocating), do the
  resample on a hop thread fed by an SPSC channel — **but only as
  a last resort**. First attempt: confirm the resampler is callback-safe.
- [x] Task A2. Unit test in `capture.rs` that drives a synthetic
  cpal-callback stand-in (a function that calls the closure 100x
  with deterministic samples) and asserts the forwarder receives
  every sample exactly once, in order, in the expected sample-rate
  domain. The test does not need a real cpal device.

#### Phase A2 — Pump SPSC bridge

- [x] Task A3. The cpal callback runs on a real-time audio thread
  and MUST NOT block on a tokio runtime. Bridge via a lock-free
  SPSC channel:
  ```rust
  // capture-side (cpal callback thread):
  let (tx, rx) = crossbeam_channel::bounded::<Vec<f32>>(64);
  capture.start_with_forwarder(move |pcm: &[f32]| {
      // bounded send; drop on overflow rather than block
      let _ = tx.try_send(pcm.to_vec());
  })?;
  ```
  `crossbeam-channel` is already a workspace dep
  (`Cargo.toml:83`); no new dependency. The bounded capacity
  prevents the audio thread from latching onto allocator pressure
  during pump backpressure; an overflow is logged at `warn!` and
  drops the frame, which is the right call (we'd rather lose a
  frame than glitch the audio thread).
- [x] Task A4. Replace the drain task at
  `crates/fono/src/session.rs:840-883` with a tokio task that
  pulls from the crossbeam `rx` and pushes into `Pump`:
  ```rust
  let drain_join = tokio::spawn(async move {
      let mut pump = pump;
      while let Ok(chunk) = tokio::task::spawn_blocking({
          let rx = rx.clone();
          move || rx.recv()
      }).await.ok().flatten() {
          // chunk: Result<Vec<f32>, RecvError> from spawn_blocking
          if let Ok(pcm) = chunk {
              pump.push(&pcm);
          } else {
              break; // tx side closed
          }
          // Honour stop signal between chunks for prompt teardown
          if drain_stop_rx.try_recv().is_ok() {
              break;
          }
      }
      pump.finish();
      drop(pump);
  });
  ```
  Implementation note: the `spawn_blocking` round-trip may add
  latency; an alternative is to use `crossbeam::channel` from a
  dedicated `std::thread` rather than tokio. Pick the simpler one
  during implementation; benchmark in Phase A4 to confirm it
  doesn't undo the latency win.
- [x] Task A5. Stop-and-flush semantics in
  `on_stop_live_dictation` (`session.rs:909`) need updating:
  - Drop the `tx` side first to signal EOF to the drain.
  - Wait for the drain to drain the channel and call `pump.finish()`.
  - Tear down the cpal stream (release the device).
  - Existing post-drain logic (`run_join.await`, transcript
    handling) is unchanged.

- [x] Task B2. Implement `StreamingStt for GroqStreaming`:
  - Spawn a tokio task that:
    1. Accumulates `StreamFrame::Pcm(chunk)` into a per-segment
       buffer.
    2. Every 700 ms (`PSEUDO_STREAM_INTERVAL`), if the buffer

## Potential Risks and Mitigations

1. **cpal callback resampler may not be realtime-safe.** The
   existing `crates/fono-audio/src/resample.rs` (`rubato`) allocates
   per process; a hard real-time audio thread should not allocate.
   **Mitigation:** if rubato allocates in the callback, do the
   resample on a hop thread fed by an SPSC channel. The hop thread
   adds maybe 5 ms of latency but keeps the audio thread clean.
   Fall-back path documented in Phase A1 Task A1 ("only as a last
   resort").

2. **`tokio::task::spawn_blocking` in Phase A2's drain task may
   negate the latency win.** Each iteration spawns a blocking task
   for the crossbeam `recv()`.
   **Mitigation:** alternative implementation using a dedicated
   `std::thread` that owns the crossbeam `rx` and cross-thread
   sends to a `tokio::sync::mpsc::UnboundedSender<Vec<f32>>` for
   the tokio side to consume. Pick the implementation in Phase A2;
   benchmark in Phase A4.

3. **Groq pseudo-stream cost surprise.** Re-POSTing the trailing
   30 s every 700 ms is roughly 43× the batch cost for a 30 s
   utterance.
   **Mitigation:** clearly document in `docs/providers.md` and in
   the wizard prompt; opt-in via `streaming: bool` (default
   `false`); ADR 0020 captures the trade-off. Future work
   (Slice B post-Wave 3) is to switch to chunked uploads of *only
   the new tail* once Groq exposes a streaming/chunked endpoint.

4. **In-flight cap drops previews on bursty audio.** A user
   speaking quickly may see preview gaps where requests piled up
   and got dropped.
   **Mitigation:** preview gaps are visually fine — the overlay
   shows the most recent stable text continuously. The
   `preview_skipped_count` metric is exposed for diagnosis;
   `fono doctor` could surface it in a follow-up commit.

5. **Cloud-mock recordings drift from the real Groq backend.**
   Groq updates its model; recordings become stale; CI passes
   while real users see different output.
   **Mitigation:** recordings are pinned to a specific model name
   (`whisper-large-v3-turbo`) and refresh manually via the
   `record-cloud` subcommand. A `last_captured` timestamp in the
   recording metadata + a future scheduled CI job that re-captures
   monthly is a good follow-up; not in scope for this slice.

6. **Hand-crafted cloud recording (Phase C1 Task C3) is a fake
   pass.** If real-key capture is skipped, the `cloud-mock` row
   is just local-Whisper output relabelled.
   **Mitigation:** explicit `notes` field in the recording calls
   it out; the gate still tests the *plumbing* (mock client wiring,
   manifest parsing, `--stt cloud-mock` flag) which is the actual
   risk. Real recordings replace the placeholder in a follow-up
   commit (small enough to stay independent).

7. **`tests/fixtures/cloud-recordings/` directory pulls in
   committed JSON the size of full Groq response bodies.** Each
   recording is ~2-5 KB; 2 recordings × 1 fixture = ~10 KB.
   Acceptable repo-size impact.

8. **`.cargo/config.toml`'s `--allow-multiple-definition` may
   interact badly with new dependencies pulled in by Thread B.**
   The Groq path doesn't add new deps (reqwest + serde already
   present), so no expected impact. If Thread B adds anything new,
   re-run the full slim+default+slim+interactive build matrix to
   confirm no link regressions.

## Alternative Approaches

1. **Skip Thread A (cpal callback push) and ship only B + C.**
   The current 30 ms polling drain works. Trade-off: the
   `live.first_partial` latency stays at ~400 ms +
   buffer-mutex-contention floor; user experience of "paints as
   you speak" is muted. Rejected — Thread A is the most
   user-visible piece of Slice B1; deferring it leaves the
   strategic plan's "biggest user-visible feature gap" claim
   unaddressed.

2. **Implement Groq streaming via WebSocket once Groq exposes
   one.** Groq doesn't have a streaming WS endpoint today
   (verified at the time of this plan; see ADR 0020). Trade-off:
   waiting for upstream blocks Slice B1 indefinitely. Rejected —
   pseudo-stream is the documented Slice B design (R4.2 of v1).
   Switch when Groq ships WS, in a follow-up commit.

3. **Implement OpenAI realtime first instead of Groq.** OpenAI
   has a real WebSocket streaming endpoint (`gpt-4o-transcribe`).
   Trade-off: OpenAI's price-per-minute is ~5× Groq's; users
   choose Groq specifically for cost. Slice B1's
   biggest-user-impact target is the existing Groq users.
   Rejected for this slice; OpenAI realtime lands as a
   follow-up plan after Slice B1.

4. **Use a real Groq backend in CI (no mock).** Trade-off:
   requires committed API keys (insecure) or scheduled-only runs
   (not a per-PR gate). Rejected — recorded mock is the v6
   plan's R18.12 design.

5. **Combine Threads A + B + C into one big commit.** Trade-off:
   smaller `git log`, but conflates capture-side work, STT-side
   work, and bench-side work; reverting one piece becomes
   awkward. Rejected — three commits is the right granularity.

6. **Defer Thread C's CI gate to Wave 4 / 5.** Trade-off: less
   churn in `.github/workflows/ci.yml` this session, but the
   cloud-mock recordings sit unverified until a later session.
   Acceptable if Thread C ends up larger than expected during
   implementation; in that case ship Tasks C1–C7 in this wave
   and defer C8 (the CI gate) to a one-task follow-up plan.
