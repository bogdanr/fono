# v0.8.0 pre-release UX corrections + cloud STT build fix

Amends the v0.8.0 release commit (`034f582`) on `main` before tagging. Five concrete fixes — four wizard-UX, one build-time bug.

## Objective

Land the five corrections requested in the post-execution review of `plans/2026-05-13-2026-05-13-wizard-catalogue-multimodal-and-multi-tts-issues-9-11-v2.md` so the wizard's first-run experience matches what was intended, and the resulting daemon actually works for a user who picks OpenAI as their primary provider.

1. Reword and re-lay-out the top-level path picker as a fixed-order two-column table.
2. Skip the language MultiSelect when the OS reports ≥ 2 detected languages.
3. Drop the "Enable live dictation?" question entirely — it's a tray-toggle, not a wizard concern.
4. Auto-enable the voice assistant on cloud paths when the primary covers both chat and TTS — no Confirm.
5. Fix the `OpenAI STT not compiled in` runtime warning by adding the cloud STT feature flags to the default `fono` build.

## Findings, ranked by user impact

1. **Fix 5 (OpenAI STT build flag) is a release blocker.** `crates/fono/Cargo.toml:77` lists `fono-stt` with **no features**, so `fono-stt`'s `default = ["groq", "wyoming"]` is all that ships. A user who picked OpenAI as primary lands in a degraded daemon on first launch. This is the loudest of the five issues and the only one that's a behavioural bug rather than UX polish.
2. **Fix 3 (drop live-dictation question)** removes the wizard's most-frequently-confused prompt for new users. Zero code risk — `config.interactive.enabled` already defaults to `false` and the tray exposes the toggle.
3. **Fix 4 (auto-enable assistant)** removes a redundant Confirm immediately after the user already committed to a cloud key that supports it. Equivalent UX delta to fix 3.
4. **Fix 1 (step-1 re-layout)** is the most-touched prompt in the whole wizard. Two-column alignment + fixed order makes it scannable instead of paragraph-shaped.
5. **Fix 2 (skip language picker on multi-detect)** removes the third prompt from a multi-language user's run; lowest individual impact, but lands alongside the others.

## Potential Risks and Mitigations

1. **`pick_interactive_mode` deletion breaks an external caller.**
   Mitigation: ripgrep the workspace for callers before deleting the function body; downgrade to `#[allow(dead_code)]` if any remain.
2. **Hard-coded column padding in Fix 1 becomes wrong when a future variant is added.**
   Mitigation: compute the padding from `items.iter().map(|s| s.0.len()).max()` instead of hard-coding.
3. **Skipping the language picker hides a legitimate user choice when the OS locale list is stale or incorrect.**
   Mitigation: the tray's Languages submenu is the documented editing surface — print a one-line `change later from the tray menu` info line on every skip.
4. **Auto-enabling the assistant might surprise users who don't want it.**
   Mitigation: an assistant with no TTS still falls through to `pick_tts_for_assistant`, where `Skip — text-only assistant` is a visible escape hatch. Local-LLM users still get the full `Confirm` flow because no catalogue primary matches their LLM backend.
5. **Adding cloud STT features balloons the slim build.**
   Mitigation: each feature flag adds only the per-provider `reqwest` reuse, no native deps; `cloud-all` is updated for slim builders, and the default is still the static binary every user already gets.
6. **`fono-stt` doesn't have features for Cartesia STT or AssemblyAI STT.** The catalogue advertises them; the wizard's `is_primary_candidate` filter excludes them (they have no LLM), but `offer_secondary_stt` lists them and would fail to build.
   Mitigation: this is pre-existing behaviour, not introduced by these fixes. File a separate follow-up ticket to either add the features or filter the secondary STT list by `cfg!(feature = "...")` reachability.

## Alternative Approaches

1. **Keep `pick_interactive_mode` but default it to off and only ask Borderline-hardware users.** Pros: preserves the hardware-aware UX. Cons: still adds a wizard prompt; the user's review was explicit — "It's not critical. The user would be able to select that later from the tray." Reject.
2. **Make the language-picker skip configurable via a CLI flag.** Pros: power-user safety. Cons: scope creep; the tray submenu already serves this need.
3. **Add Cartesia/AssemblyAI STT features in this same commit to keep `offer_secondary_stt` rigorous.** Pros: completeness. Cons: requires writing two more STT clients, not just feature flags — those clients don't exist yet. Defer to a follow-up issue.
4. **Tag v0.8.0 first, then ship these fixes as v0.8.1.** Pros: cleaner release ledger. Cons: v0.8.0 is broken for OpenAI users out of the gate (Fix 5 is a behavioural bug); shipping a known-broken release just to tag it is hostile.
